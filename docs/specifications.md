## 1. ОБЩИЕ ПОЛОЖЕНИЯ

### 1.1. Наименование программы
Серверная часть прикладной программы «Система управления базой данных железнодорожного вокзала» (RWSDBv2.1 - Railway Station Database System v2.1).

### 1.2. Область применения
Программа предназначена для управления данными железнодорожного вокзала, включая информацию о сотрудниках, поездах, расписании, пассажирах, билетах и сервисах.

### 1.3. Назначение разработки
Серверная часть обеспечивает:
- Хранение и обработку данных в СУБД PostgreSQL
- Предоставление HTTP API для клиентских приложений
- Разграничение прав доступа между пользователями
- Резервное копирование базы данных

## 2. ТРЕБОВАНИЯ К ПРОГРАММЕ

### 2.1. Требования к функциональности

#### 2.1.1. Серверная часть должна предоставлять следующие операции:
- Просмотр данных таблиц базы данных
- Фильтрация содержимого таблиц
- Добавление записей в таблицы
- Обновление записей в таблицах
- Удаление записей из таблиц
- Выполнение специальных запросов (SELECT)
- Создание резервных копий базы данных
- Сохранение результатов запросов в файл (JSON, CSV, Excel)

#### 2.1.2. Архитектура взаимодействия
- Серверная часть реализуется в виде HTTP-сервера
- Тела запросов и ответов представляются в формате JSON
- Для взаимодействия с ресурсами используются стандартные HTTP-методы:
  - GET – получение данных о ресурсе
  - POST – создание нового ресурса
  - PUT – обновление существующего ресурса
  - DELETE – удаление ресурса

### 2.3. Требования к ролевой модели

#### 2.3.1. Роли пользователей
В системе выделяются две роли:

**1. Пользователь (User)**
- Право просмотра всех таблиц (GET-запросы)
- Право сохранения результатов запросов
- Право редактирования всех таблиц, кроме справочных
- Право добавления записей в операционные таблицы
- Право удаления записей из операционных таблиц
- НЕ может редактировать справочные таблицы
- НЕ может создавать резервные копии БД

**2. Суперпользователь (Superuser/Admin)**
- Все права обычного пользователя
- Право редактирования справочных таблиц (positions, subjects, lesson_types)
- Право создания резервных копий базы данных
- Для выполнения действий от имени суперпользователя приложение должно запрашивать пароль

#### 2.3.2. Идентификация справочных таблиц (LookUp Tables)
К справочным таблицам относятся:
- positions – должности сотрудников
- subjects – учебные предметы/дисциплины
- lesson_types – типы занятий

Справочные таблицы содержат нормативно-справочную информацию, которая изменяется редко.
Запись в справочные таблицы разрешена только суперпользователю.

#### 2.3.3. Механизм аутентификации суперпользователя
- Аутентификация осуществляется через HTTP-заголовок Authorization
- Формат: `Authorization: Bearer {token}`
- Token = SHA256 хэш от пароля суперпользователя
- Пароль суперпользователя по умолчанию: «4444»
- Пароль хранится в конфигурационном файле config.json

### 2.5. Требования к скриптам базы данных

#### 2.5.1. Скрипт создания базы данных (postgres_create_schema.sql)
- Создание всех таблиц с указанными выше полями и типами данных
- Создание всех внешних ключей (FK)
- Создание индексов для часто используемых полей
- Создание представлений (views) для сложных запросов

#### 2.5.2. Скрипт наполнения справочников (postgres_populate_lookup_tables.sql)
- Наполнение таблицы positions (не менее 10 записей)
- Наполнение таблицы subjects (не менее 10 записей)
- Наполнение таблицы lesson_types (не менее 5 записей)

#### 2.5.3. Скрипт генерации тестовых данных (postgres_generate_main_data.sql)
- Генерация данных для таблицы employees (не менее 100 записей)
- Генерация данных для таблицы students (не менее 200 записей)
- Генерация данных для таблицы groups (не менее 20 записей)
- Генерация данных для таблицы marks (не менее 500 записей)
- Генерация данных для таблицы lessons (не менее 100 записей)
- Генерация данных для остальных таблиц согласно схеме

#### 2.5.4. Скрипт функций и триггеров (postgres_functions_triggers.sql)
- Функция для автоматического обновления даты изменения записи
- Триггер для ведения журнала изменений
- Функция для проверки целостности данных
- Триггер для каскадного удаления связанных записей

#### 2.5.5. Скрипт резервного копирования (postgres_backup.sql)
- Функция создания резервной копии базы данных
- Функция восстановления из резервной копии

### 2.6. Требования к техническим средствам
- Сервер: ПК с ОС Windows/Linux/macOS
- СУБД: PostgreSQL версии 10 или выше
- Библиотека: libpq (через psycopg2 для Python)
- Сеть: поддержка HTTP-протокола

### 2.7. Требования к программной среде
- Язык программирования: Python 3.8 или выше
- Библиотеки:
  - psycopg2-binary >= 2.9.0 (для работы с PostgreSQL через libpq)
  - openpyxl >= 3.0.0 (для экспорта в Excel)
  - PyQt6 >= 6.4.0 (опционально, для GUI клиента)

## 3. ТРЕБОВАНИЯ К НАДЕЖНОСТИ

### 3.1. Требования к целостности данных
- Все транзакции должны фиксироваться или откатываться при ошибках
- Внешние ключи должны обеспечивать ссылочную целостность
- Триггеры должны предотвращать некорректные изменения данных

### 3.2. Требования к безопасности
- Пароль суперпользователя должен храниться в хешированном виде
- Все SQL-запросы должны использовать параметризацию для защиты от SQL-инъекций
- Доступ к справочным таблицам должен быть ограничен для обычных пользователей

## 4. ТРЕБОВАНИЯ К ЭКСПЛУАТАЦИИ

### 4.1. Запуск сервера
```bash
python -m RWSDBv2.1
# или
python server.py
```

### 4.2. Конфигурация
Конфигурационный файл config.json должен содержать:
```json
{
  "admin_password": "4444",
  "pg_host": "localhost",
  "pg_database": "railway_station",
  "pg_user": "postgres",
  "pg_password": "postgres",
  "pg_port": 5432
}
```

### 4.3. Тестирование
Тестирование сервера осуществляется с помощью:
- psql для выполнения SQL-скриптов
- curl или браузера для тестирования HTTP API
- pytest для автоматизированного тестирования

## 5. ПРИЛОЖЕНИЯ

### 5.1. Примеры HTTP-запросов

#### GET-запрос на получение всех сотрудников
```
GET http://localhost:8080/api/employees
```

#### GET-запрос с фильтрацией
```
GET http://localhost:8080/api/employees?position=Station Manager&limit=10
```

#### POST-запрос на создание записи (обычный пользователь)
```
POST http://localhost:8080/api/employees
Content-Type: application/json

{
  "full_name": "Иванов Иван Иванович",
  "position": "Engineer",
  "experience": 5
}
```

#### POST-запрос на создание записи в справочник (суперпользователь)
```
POST http://localhost:8080/api/positions
Content-Type: application/json
Authorization: Bearer 4444

{
  "name": "Senior Engineer"
}
```

#### PUT-запрос на обновление записи
```
PUT http://localhost:8080/api/employees/1
Content-Type: application/json

{
  "experience": 6
}
```

#### DELETE-запрос на удаление записи
```
DELETE http://localhost:8080/api/employees/1
```

#### POST-запрос на создание резервной копии (суперпользователь)
```
POST http://localhost:8080/api/backup
Authorization: Bearer 4444
```
